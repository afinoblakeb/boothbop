# BoothBop 0.0.4 Build 0.0.5: Media Quality Foundation

## Release

- Candidate version: App Store `0.0.4`, replacement build `0.0.5`
- Owner: BoothBop
- Decision date: 2026-07-22
- Target evaluation date: 2026-07-22

## Consumer Problem

Creating, editing, reopening, and sharing a BoothBop session can feel slow and
can silently reduce the quality of the photos the user just captured.

## Evidence

- User feedback: First GIF sharing feels like the app has stalled; Low output is
  visibly unacceptable; shared video looks materially softer than the strip.
- Product observation: The shipping path builds a 7.9-megapixel preview on the
  UI thread, JPEG-encodes native video frames before H.264, stores gallery
  frames at JPEG quality 0.85, and reopens every saved frame at 720px.
- Frequency or severity: These defects touch the core capture-to-share journey
  and affect every session, not an edge feature.
- Why this problem matters now: Social sharing is the principal completion
  action and the product promises private, beautiful media created on device.

## Proposed Capability

Replace the current ad-hoc media work with one high-quality render pipeline.
BoothBop always captures and exports at its best supported quality, uses small
screen previews, preserves source dimensions in My Photos, deduplicates and
prewarms share renders, keeps progress feedback responsive, and encodes native
video without an intermediate lossy image format.

This is a quality correction, not a new creative feature.

## Simpler Alternatives Considered

1. Increase only the video bitrate. Rejected because it cannot recover pixels
   already lost to JPEG, gallery downscaling, or low-resolution capture.
2. Add only a loading spinner. Rejected because synchronous work freezes the
   spinner and leaves the destructive media pipeline unchanged.

## Non-Goals

- New templates, props, captions, accounts, analytics, subscriptions, or social
  network SDKs.
- Direct publishing to Instagram or destination-specific authentication.
- A new editor or a broader feature release.

## User Journey

The user captures four photos and reaches review without a post-capture stall.
The visible strip and editor use a screen-sized preview that updates within one
interaction frame. Selecting GIF immediately displays the GIF while BoothBop
prepares the Instagram-safe MP4 once in the background. The toolbar shows a
small, responsive preparation state until the share asset is ready. Tapping
Share either opens immediately or waits on that same in-flight job; it never
starts a duplicate encode. Saved sessions reopen at their original dimensions.

## Required States

- First use: Best quality is automatic; no quality configuration is required.
- Repeat use: Prepared media is reused until an output-affecting choice changes.
- Loading: Visible, responsive, accessible progress appears within 100ms.
- Empty: Export actions remain unavailable before four valid frames exist.
- Error: Preparation failure is actionable and the original GIF remains usable.
- Permission denied: Capture and Photos failures preserve the existing recovery.
- Offline: Capture, rendering, gallery, saving, and sharing remain on device.
- Compact phone: Preview, progress, and actions remain visible at 320x568.
- Large phone: Preview uses available space without rendering excess pixels.
- Accessibility: Busy state is announced and controls expose disabled/readiness
  state without relying on animation or color.

## Success Criteria

- Behavioral: One keyed job serves preview, autosave, save, and share callers;
  stale work cannot replace current media.
- Quality: Gallery round-trips preserve source dimensions; native video receives
  lossless frame inputs; H.264 uses High Profile at 1080x1920 for social export;
  no legacy Low or Standard preference can reduce a new export.
- Performance: Screen preview cell size is at most 360px; filter interaction
  produces no task over 100ms in the production browser journey; share feedback
  appears within 100ms; a prewarmed native share opens without a new encode.
- TestFlight feedback: App Store-upgrade install shares a sharp Post, Reel, and
  Story; first Share never appears frozen; reopened sessions match fresh output.

## Removal Criteria

Hold the replacement if it lowers decoded media fidelity, adds a sustained
black/blank state, exceeds the performance budgets, loses a gallery session, or
causes the social MP4 to fail Instagram import.

## Test Plan

- Unit coverage: Quality migration, source-size preservation, render-job
  deduplication/invalidation, profile values, social geometry, frame ordering.
- Browser journey: Long-task budgets, responsive busy state, prewarm reuse,
  editor latency, full-resolution export dimensions, gallery round-trip.
- Native simulator: Clean and update launch, capture review, background prewarm,
  share-sheet presentation, cancellation, compact/standard/large layouts.
- Media inspection: Decode PNG/GIF/MP4, compare dimensions and representative
  pixels, verify H.264 profile/container/duration/no audio and feed-safe geometry.
- Physical update install: Install build 0.0.5 through TestFlight over 0.0.3 and
  share to Instagram Post, Reel, Story, Messages, and Save Video.
- Airplane Mode: Complete capture, edit, gallery reopen, and every export.

## Prototype Reuse

- `src/lib/videoNative.ts` and `BoothBopVideo` are corrected, not replaced by a
  prototype branch.
- `src/lib/session.ts`, `src/lib/socialShare.ts`, and the 0.0.4 crop contract are
  retained.
- No wholesale merge from a prototype branch is permitted.
