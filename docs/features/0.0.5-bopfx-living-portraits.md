# BoothBop Feature Brief: BopFX Living Portraits

## Release

- Candidate version: `0.0.5`
- Owner: Blake Billiet
- Decision date: July 23, 2026
- Target evaluation date: July 30, 2026

## Consumer Problem

BoothBop's existing color looks do not transform a four-photo session into
distinctive, animated media that people immediately want to share.

## Evidence

- User feedback: The owner explicitly wants effects that go beyond generic
  color changes and use the iPhone's native face and graphics capabilities.
- Product observation: The four-frame capture, GIF, Boom, and social-video
  outputs already create a natural canvas for motion-driven portrait art.
- Frequency or severity: Every completed session reaches the same review and
  export surface, so a strong creative effect can improve the core loop rather
  than add a side workflow.
- Why this problem matters now: Capture, lifecycle, media quality, and sharing
  have been hardened enough to support one ambitious creative capability
  without knowingly building on an unstable foundation.

## Proposed Capability

Add **BopFX: Living Portraits**, a native, non-destructive face-aware effects
engine. The user can select a small set of dramatic effects from the camera,
see the selected effect in the live preview, capture the normal four-photo
sequence, change or remove the effect in review, and export the same visual
result as a strip, GIF, Boom, or social video.

Four concepts enter the prototype:

1. **Spectral Echo** duplicates faces or foreground subjects with chromatic
   offsets that phase across the four captured frames.
2. **Funhouse** applies smooth, face-landmark-centered geometric distortion
   without indiscriminately warping the full image.
3. **Cutout Chorus** isolates people and arranges repeated silhouettes into a
   rhythmic graphic composition.
4. **Mirror Bloom** builds a recognizable face-centered reflection pattern
   rather than a generic full-frame kaleidoscope.

The evaluation build may contain all four. The release contains only effects
that meet every quality gate, with a target of two or three excellent effects.
The capability ships as one BopFX picker, not as separate feature additions.

## Simpler Alternatives Considered

1. Add more Core Image color filters. Rejected because color recipes do not
   solve the stated desire for distinctive spatial or animated output.
2. Build a full Animoji-style avatar system. Rejected because face tracking is
   only the input; producing excellent rigged characters requires a separate
   art and animation pipeline and would weaken BoothBop's photographic identity.
3. Run the whole camera through ARKit. Rejected as the default architecture
   because BoothBop must preserve full-resolution `AVCapturePhotoOutput`
   capture, native autofocus, exposure, white balance, and responsive shutter
   behavior. ARKit remains an optional enhancement only if a spike proves it
   can coexist without degrading those invariants.

## Non-Goals

- No avatars, masks, props, virtual makeup, or general AR scene editor.
- No cloud processing, generative AI, accounts, uploads, or face identity
  storage.
- No attempt to recognize a person, infer sensitive traits, or persist face
  geometry.
- No large effects catalog, effect marketplace, paywall, or subscription in
  this release.
- No replacement of the classic unfiltered BoothBop experience as the default.

## User Journey

The camera opens exactly as it does today. A single effects control opens a
bottom carousel above the shutter controls. Each item uses a real live preview,
with `Original` first and the current effect clearly selected. Choosing an
effect immediately updates the preview without starting a separate mode.

The user closes the carousel or taps the shutter and completes the normal
3-2-1, four-photo sequence. Countdown duration, shutter freeze, and recovery
buffer remain unchanged. Review shows the completed strip and exports with the
same effect applied. The Edit toolbar exposes the same BopFX picker so the user
can switch effects or return to `Original` without recapturing.

Raw captured masters remain unchanged. Save and Share remain in their current
locations. If an effect cannot render, BoothBop preserves the session, falls
back to `Original`, and explains the recovery without blocking export.

## Required States

- First use: `Original` remains selected; no tutorial or permission is added.
- Repeat use: The last deliberate BopFX selection may persist, with an obvious
  path back to `Original`.
- Loading: A newly selected effect shows bounded progress without freezing the
  camera or disabling navigation.
- Empty: The picker always contains `Original`; unavailable effects are omitted
  rather than presented as disabled promises.
- Error: Rendering failure restores `Original` and retains all raw photos.
- Permission denied: Existing camera recovery remains unchanged; BopFX requires
  no permission beyond camera access.
- Offline: Every effect and asset works entirely on device.
- Compact phone: The carousel and shutter remain visible without vertical
  scrolling.
- Large phone: Preview and controls remain visually connected rather than
  spreading into unused space.
- Accessibility: Controls have names, values, selected state, Dynamic Type
  tolerance, sufficient contrast, and no motion-only meaning.

## Success Criteria

- Behavioral: Preview, all four captures, review, Retake One, gallery reload,
  strip, GIF, Boom, and social video agree on the selected effect.
- Behavioral: Original masters survive effect changes, failures, app
  backgrounding, and relaunch.
- Quality: At least two prototypes are visibly transformative, recognizable,
  and judged worth sharing by the owner on a physical iPhone.
- Quality: Full-resolution output has no landmark drift, jagged masks, edge
  halos, accidental clipping, or preview-to-export composition surprise.
- Performance: Selecting no effect adds no meaningful work to launch or the
  classic capture path.
- Performance: On the paired iPhone 15 Pro, an active effect sustains a
  responsive live preview, never delays the selected countdown, and does not
  delay review behind avoidable synchronous rendering.
- Test feedback: The owner and a small group of friends can select, capture,
  understand, and share BopFX output without instructions.

## Removal Criteria

Hold the capability if fewer than two effects feel substantially more valuable
than the existing looks, if preview and exported output cannot match, if
capture reliability regresses, if clean masks require cloud processing, or if
the effect makes faces look accidentally broken rather than intentionally
stylized.

## Test Plan

- Unit coverage: Effect descriptors, capability selection, coordinate
  transforms, landmark smoothing, phase progression, fallback selection, and
  deterministic render recipes.
- Browser journey: Picker interaction, non-destructive review changes,
  persistence, compact layout, export invalidation, and failure fallback using
  deterministic native mocks.
- Native simulator: Lifecycle, navigation churn, unsupported-capability
  fallback, layout, and sample-image rendering. Do not claim live camera,
  Vision, or Metal proof from Simulator alone.
- Physical clean install: Cold launch, first selection, four-shot capture,
  preview/export alignment, Retake One, every output type, and share sheet.
- Physical update install: Existing gallery/settings preservation and first
  BopFX use after upgrading.
- Airplane Mode: Complete capture, review, gallery, and all exports without a
  network.

## Prototype Reuse

- Start from the native camera invariants at release tag
  `appstore-v0.0.4-build-0.0.5`.
- Reuse the current `AVCaptureVideoDataOutput`, `AVCapturePhotoOutput`,
  TrueDepth camera preference, Core Image context, and serialized camera
  ownership in `ios/App/App/AppDelegate.swift`.
- Reuse no visual effect code from the June prototype branches without a
  line-by-line review.

## Technical Spike

1. Keep the existing `AVCaptureSession`; do not let ARKit replace the native
   full-resolution still pipeline.
2. Use stable Vision face landmarks as the broad-device geometry source.
3. Evaluate person segmentation and TrueDepth depth data as optional inputs,
   guarded by runtime capability checks.
4. Replace the standard preview layer only inside the BopFX path with an
   `MTKView` renderer modeled on Apple's AVCamFilter architecture.
5. Keep capture buffers in a native YUV format and render through Core Image or
   Metal; do not request BGRA camera buffers merely for convenience.
6. Analyze preview frames at a bounded cadence and smooth geometry between
   observations. Render every preview frame without requiring Vision to finish
   every frame.
7. Re-run the same effect descriptor at full captured-photo resolution. Never
   upscale the preview as an exported master.
8. Keep all native feature calls behind a narrow TypeScript adapter and a
   mockable Capacitor contract.
